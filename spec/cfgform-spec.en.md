# `.cfgform` Sidecar Metadata Specification v2.1

> **Single Source of Truth**. The developer-side `prep-tool` **generates** per this spec; the user-side `configurator` **consumes** per this spec. Any code change must first update this document.
>
> v2.0 core change: upgraded from "JSON only" to a **format-agnostic universal config comprehension layer**. Introduced **format adapters** and the **format-neutral suffix `.cfgform`**; added secret masking, Dry-run preview, conditional validation, profiles, defaults/reset, read-only locking, and built-in schema library.

---

## 0. Product Positioning (Why This Exists)

| Dimension | Description |
| --- | --- |
| **Real pain point** | Config files universally lack clarity of author intent: beginners can't read them, veterans find them messy, and JSON/YAML/.env are extremely prone to errors from brackets/indentation/BOM/quotes. |
| **Product essence** | A **universal config comprehension & safe editing layer**. File format is merely a serialization shell; the scarce value is the **semantic layer (structural constraints + plain language + author intent)**. |
| **Moat** | `.cfgform` sidecar (schema + ui + meta), format-agnostic, reusable, shareable across ecosystems. |
| **Two audiences** | Developers (use `prep-tool` to generate sidecar once); End users (use `configurator` to double-click for safe editing, zero LLM, zero awareness). |

---

## 1. Design Iron Rules

1. **Format-agnostic core**: Rendering/validation/audit/backup are identical across all formats; format differences are fully isolated within "format adapters."
2. **No pollution, no intrusion**: All metadata is merged into a **single `.cfgform` file** (recognized only by this ecosystem); regular users never accidentally touch it.
3. **Original file keeps its original name and form**: After editing, **write back to the original file itself**, never rename; auto-backup before write.
4. **Lossless write-back first**: Exert maximum effort to preserve the original file's **comments, key order, format** (see В§6 for per-format fidelity).
5. **Full audit trail + data transparency**: What was read / what was written / where backed up / which field changed from X to Y / which model was invoked / which source files were read вЂ” all written as human-readable audit log; **secrets never enter logs, never enter sidecar, never appear in preview in cleartext**.

## 2. File Naming Convention (v2.0: Switched to "Append" Pairing, Eliminating Ambiguity)

| Role | Naming Rule | Example |
| --- | --- | --- |
| Target config (user edits, target app consumes) | Original name, never renamed | `config.json` / `.env` / `app.toml` / `docker-compose.yml` |
| **Sidecar metadata** (this spec) | **Target full filename + `.cfgform`** | `config.json.cfgform` / `.env.cfgform` / `docker-compose.yml.cfgform` |
| Backup | Target name + UTC timestamp + `.bak` | `config.json.20260620T101530Z.bak` |
| Audit log | Fixed name in same directory | `cfgform-audit.log` |

> **Why "append" rather than "replace extension"**: Replacing the extension would cause `config.json` and `config.yaml` sidecars to collide; appending suffix is globally unique, format-agnostic, and `configurator` can obtain the target filename simply by stripping `.cfgform` from `*.cfgform`.
>
> **Backward compatibility**: `configurator` still recognizes legacy `xxx.jsonform` (treated as `format=json`), but `prep-tool` only produces new-style `.cfgform`.

## 3. `.cfgform` File Structure (Always JSON)

```jsonc
{
  "$cfgform": "2.0",                    // Spec version, required
  "target": "docker-compose.yml",       // Target filename (same dir as sidecar), required
  "format": "compose",                  // json|env|toml|yaml|ini|compose, required (see В§5)
  "title": "Service Orchestration Config",
  "schema": { /* JSON Schema draft-07: structure & hard constraints */ },
  "ui": { /* RJSF uiSchema + extensions of this spec (see В§4) */ },
  "profiles": {                          // Optional, multi-environment (see В§4.4)
    "active": "dev",
    "list": ["dev", "staging", "prod"]
  },
  "meta": {
    "generatedBy": "prep-tool/2.0.0",
    "generatedAt": "2026-06-20T10:15:30Z",
    "stackDetected": "node",
    "llm": { "used": true, "model": "deepseek-chat",
             "note": "Types inferred from original file + source/README refined by LLM; constraints & help text generated by LLM вЂ” authors please review." },
    "sources": ["README.md", "src/config.ts"]
  }
}
```

> `meta.format` is also mirrored at the top-level `format` (top-level is authoritative). `schema`/`ui` are always based on the **canonical data tree** (the format-neutral JSON after adapter parsing), independent of the target file's physical format.
>
> **On version numbers**: this document's **revision** is v2.1 (see the changelog at the end); whereas the on-disk field `$cfgform` is the **format-compatibility version**, currently still `"2.0"`. The two are intentionally distinct вЂ” a document revision need not bump the sidecar's compatibility version, so do not write `$cfgform` as `"2.1"`.

## 4. `ui` Extensions (New Additions on Top of RJSF uiSchema)

### 4.1 Standard RJSF
`ui:widget`(text/textarea/password/url/email/range/radio/select/checkbox/color/date), `ui:placeholder`, `ui:help` (plain-language hint), `ui:options`, `ui:enumNames` (human-readable enum labels), `ui:order`, `ui:collapsible`/`ui:collapsed`.

### 4.2 `ui:secret` (Secret Masking, Safety-Critical)
When a field is marked `"ui:secret": true`: в‘  Displayed masked as `вЂўвЂўвЂўвЂўвЂўвЂў` by default, with a "Show/Hide" toggle; в‘Ў **The field value is never written to the audit log** (log only records `Field X: modified (secret, value not recorded)`); в‘ў In Dry-run preview the value is **masked by default**, can be manually temporarily revealed with a prominent warning; в‘Ј Backup files still contain the real value (local files, protected by directory permissions). `prep-tool` auto-suggests `ui:secret` for common secret key names (key/token/secret/password/dsn/credentialвЂ¦).

### 4.3 `ui:readOnly` (Author-Locked Read-Only)
When a field is marked `"ui:readOnly": true`, the form control is greyed out and non-editable, annotated "Author Locked." Used for fields the author does not want users to modify (e.g., internal version numbers).

### 4.4 Profiles (Multi-Environment) вњ… Implemented
Top-level `profiles = { active, list:[...], overrides:{ <name>:{ <dot-path>: value } } }`. `configurator` provides an environment switcher at the top; effective data = base target data вЉ• `overrides[active]`, using base as foundation with **non-cumulative** overlay on switch. Save: effective values go through the unified pipeline and are written back to the single target file; an additional "Save current changes as [active] overrides" button writes the diff (current vs base) into the sidecar `profiles.overrides[active]` and saves the sidecar (audited).

### 4.5 Defaults / Diffs / Reset
`default` in `schema` is used for: в‘  Populating newly added fields; в‘Ў Highlighting fields in the form that "differ from default"; в‘ў A per-field "Reset to default" button.

### 4.6 Conditional / Cross-Field Validation
Directly uses JSON Schema native `if/then/else`, `dependencies`, `allOf`, executed by AJV and rendered by RJSF. Example: when `mode=prod`, `https` is required. `prep-tool`'s LLM prompt should produce such conditional constraints when inferrable.

## 5. Format Adapter Contract

Each format implements one adapter, **solely** responsible for "file text в‡„ canonical data tree"; the rest of the core remains unchanged:

```
Text(bytes) в”Ђв”Ђparseв”Ђв”Ђв–¶ Canonical data tree(serde_json::Value) в”Ђв”Ђв–¶ schema+ui в”Ђв”Ђв–¶ RJSF form в”Ђв”Ђв–¶ Edit
Canonical data tree в”Ђв”Ђserialize(preserve comments/order/secrets/format)в”Ђв”Ђв–¶ Text(bytes) в”Ђв”Ђatomic writeв”Ђв”Ђв–¶ Target file
```

| `format` | Target File Example | Parser / Method | Notes |
| --- | --- | --- | --- |
| `json` | `config.json` | serde_json | Implemented (v1). |
| `env` | `.env` | Custom line-level parser | KV, preserves comments/order, quoted/multiline, `ui:secret` common. |
| `toml` | `app.toml`,`pyproject.toml` | `toml_edit` | **Surgically lossless**, preserves comments/order. |
| `yaml` | `*.yml/*.yaml` | `serde_yaml`(+local rewrite) | Fidelity see В§6; comment preservation is a known difficulty. |
| `ini` | `*.ini/*.properties/*.conf` | Custom section/KV | Section + KV, preserves comments/order. |
| `compose` | `docker-compose.yml` | YAML adapter + **built-in compose schema** | LLM-free, out-of-the-box (built-in schema library). |

## 7. Two-Side Behavioral Contract

### 7.1 `prep-tool` (Developer Side, Generation)
1. Select target config file в†’ determine **format** by extension/content в†’ detect tech stack (`meta.stackDetected`).
2. Parse with corresponding adapter в†’ canonical data tree в†’ infer structural baseline schema (universal across formats).
3. Call LLM (OpenAI compatible, reads README/source) в†’ refine types, generate constraints/enum labels/help text/`ui:secret`/conditional constraints в†’ synthesize `schema`+`ui`.
4. **Inject and log**: write `<target>.cfgform`; **do not modify original file**; write `cfgform-audit.log` (which files read, which model used, which sidecar written).
5. LLM credential priority: **UI temporary input (memory only) > system env vars > install-dir `.env`**; secrets never written to disk/log/sidecar.

### 7.2 `configurator` (User Side, Consumption, No LLM)
1. On startup, scan the directory for `*.cfgform` (compatible with legacy `*.jsonform`).
2. For each sidecar: read target file with `format` adapter в†’ canonical data tree в†’ RJSF (schema+ui) rendering в†’ AJV real-time validation в†’ human-readable Chinese errors, required-field red star, secret masking, read-only lock, default-value diff highlighting.
3. **Save (two-step)**: First **Dry-run preview** вЂ” serialize to target format text + diff against current file (secrets masked by default) в†’ user confirms в†’ backup original file в†’ **atomic write** (tempв†’rename) to prevent corruption в†’ per-field audit log (secrets not recorded).
4. Directory has target file but no `.cfgform`: friendly prompt "Missing form description file; please ask the author to generate one with prep-tool."

## 8. Security Red Lines
- Always backup before write; atomic write to prevent interruption corruption; UTF-8 no BOM, `\n` line endings (env/ini/yaml preserve original line-ending policy).
- Secrets: `ui:secret` fields never enter logs, never enter sidecar, preview masked by default; LLM Key never hardcoded/written to disk/logged; `.env` in `.gitignore`.
- Audit log records only filenames/field paths/non-secret value changes.
